# Candidate Dashboard

Тестовое Flutter-приложение для просмотра кандидатов. В нём можно открыть
список, найти кандидата по имени, отфильтровать по вердикту, отсортировать
кандидатов, открыть подробную карточку и поменять статус кандидата.

Я также добавил кэш, чтобы список открывался без сети, и локальное сохранение
статусов, чтобы изменения не терялись после перезапуска приложения.

## Запуск

```bash
flutter pub get
dart run build_runner build --delete-conflicting-outputs
flutter run
```

Проверки:

```bash
flutter analyze
flutter test
```

## Что есть в проекте

- список кандидатов;
- поиск по ФИО с debounce;
- фильтр по verdict;
- сортировка по имени, опыту и дате;
- пагинация списка;
- loading, empty и error состояния;
- экран деталей кандидата;
- обработка случая, когда кандидат не найден;
- изменение статуса с optimistic update;
- откат статуса при ошибке сервера;
- кэш списка кандидатов;
- сохранение изменённых статусов локально;
- очередь синхронизации статусов после офлайна;
- индикация офлайн-режима;
- светлая и тёмная тема;
- адаптивная верстка для телефона и планшета;
- тесты для основной логики.

## Архитектура

Я выбрал упрощённую структуру:

```text
lib/
  core/
  data/
  features/
```

В `core` лежат общие вещи: тема, роутинг, DI, базовые виджеты и helper-ы.
В `data` лежат модели, datasource-ы и repository.
В `features` лежат экраны списка и деталей.

Основной поток данных:

```text
Dio -> CandidateApiService -> CandidateRepository -> CandidatesListBloc -> UI
```

`CandidateRepository` отвечает за загрузку данных, кэш, офлайн-режим,
локальные статусы, optimistic update, rollback и очередь синхронизации.

`CandidatesListBloc` отвечает за состояние списка и деталей: загрузку,
фильтрацию, поиск, пагинацию, выбранного кандидата и процесс изменения статуса.
Один BLoC используется для списка и деталей, поэтому статус кандидата остаётся
одинаковым на обоих экранах.

## Почему без domain-слоя

Я не стал добавлять отдельный `domain`-слой, потому что в этом проекте он почти
ничего бы не делал. Use case-и получились бы просто обёртками над repository:

```text
GetCandidatesUseCase -> repository.fetchCandidates()
UpdateStatusUseCase -> repository.updateStatus()
```

Из-за этого файлов стало бы больше, но логика не стала бы понятнее. Поэтому я
оставил слой `data` + `features`, а бизнес-логику разделил между repository и
BLoC.

Если бы проект был больше, я бы добавил `domain`-слой. Например, если бы
появились сложные правила отбора кандидатов, несколько источников данных,
несколько реализаций repository или отдельные сценарии, которые нужно
переиспользовать в разных местах приложения.

## Выбранные решения

- `flutter_bloc` - для состояния списка и деталей.
- `go_router` - для навигации и маршрута `/candidate/:id`.
- `dio` - для сетевого слоя и mock backend через interceptor.
- `hive` - для кэша, локальных статусов, очереди синхронизации и темы.
- `freezed` + `json_serializable` - для моделей и JSON.
- `get_it` - для dependency injection.
- `bloc_test` и `mocktail` - для тестов.

## Компромиссы

- Mock backend сделан через Dio interceptor и assets.
- Синхронизация статусов после офлайна простая: применяется последний локально
  выбранный статус.
- Conflict resolution для статусов не делал.
- Большой список лежит в `assets/data/candidates-large.json`, но по умолчанию
  используется обычный файл `assets/data/candidates.json`.
- Тестами покрыты основные сценарии, но не все возможные edge cases.

## Что не успел

- Сделать полноценный conflict resolution для офлайн-синхронизации.
- Добавить отдельный экран/панель для просмотра очереди синхронизации.
- Довести покрытие тестами до production-уровня.
- Добавить больше мелких UI-анимаций.
